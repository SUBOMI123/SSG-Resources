  <action>
Create .github/workflows/test.yml for running tests on every push:

```yaml
name: Test

on:
  push:
    branches:
      - main
      - develop
  pull_request:
    branches:
      - main
      - develop

jobs:
  test:
    runs-on: ubuntu-latest
    
    services:
      postgres:
        image: postgres:17-alpine
        env:
          POSTGRES_USER: ssg_user
          POSTGRES_PASSWORD: ssg_password
          POSTGRES_DB: ssg_test
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
        ports:
          - 5432:5432

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Setup database schema
        run: npx prisma migrate deploy
        env:
          DATABASE_URL: postgresql://ssg_user:ssg_password@localhost:5432/ssg_test

      - name: Run tests
        run: npm test
        env:
          DATABASE_URL: postgresql://ssg_user:ssg_password@localhost:5432/ssg_test
          NODE_ENV: test

      - name: Upload coverage
        uses: codecov/codecov-action@v3
        with:
          files: ./coverage/lcov.info
        if: always()
```

This workflow:
- Runs on every push to main/develop and all PRs
- Spins up PostgreSQL 17 for database tests
- Runs Prisma migrations to set up schema
- Executes `npm test`
- Blocks merge if tests fail

Per D-04: Atomic test-before-deploy workflow.

  <name>Task 2: Create Deploy GitHub Actions Workflow</name>
  <files>.github/workflows/deploy.yml</files>
  <action>
Create .github/workflows/deploy.yml for automatic Vercel deployment on main merge:

```yaml
name: Deploy

on:
  push:
    branches:
      - main

jobs:
  deploy:
    runs-on: ubuntu-latest
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Deploy to Vercel
        uses: vercel/action@v29
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
          vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}
          production: true  # Deploy to production URL, not preview

      - name: Comment on PR
        uses: actions/github-script@v7
        if: github.event_name == 'pull_request'
        with:
          script: |
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: '✅ Deployed to Vercel! [View live →](https://your-app.vercel.app)'
            })
```

This workflow:
- Runs only on main branch (after PR merge)
- Uses Vercel CLI to deploy
- Deploys to production URL (not preview)
- Requires GitHub secrets for Vercel authentication

Per D-04: Zero-touch production deployments after tests pass.

Follow these steps to enable Vercel deployment from GitHub Actions:

1. **Get Vercel Token:**
   - Visit https://vercel.com/account/tokens
   - Click "Create Token"
   - Name: "GitHub Actions"
   - Scope: Full Account
   - Copy the token (you'll use it once)

2. **Get Vercel Project IDs:**
   - Visit your Vercel project settings
   - Copy the "Project ID" from dashboard
   - Copy the "Org ID" (visible in Settings → General)

3. **Add GitHub Secrets:**
   - Go to your GitHub repository
   - Settings → Secrets and variables → Actions
   - Click "New repository secret"
   - Add three secrets:
     - Name: `VERCEL_TOKEN`, Value: (paste token from step 1)
     - Name: `VERCEL_ORG_ID`, Value: (paste org ID from step 2)
     - Name: `VERCEL_PROJECT_ID`, Value: (paste project ID from step 2)
   - Click "Add secret" for each

4. **Set Branch Protection Rules (Optional but Recommended):**
   - Go to Settings → Branches → Branch protection rules
   - Add rule for "main" branch
   - Require:
     - "Require status checks to pass before merging"

   - This prevents merging if tests fail

   - Watch Actions tab — should see "Test" workflow run

```markdown
# Deployment Pipeline

This project uses GitHub Actions for automated testing and deployment.

## Workflows

### 1. Test Workflow (test.yml)
Triggered on: Every push and pull request

Steps:
1. Checkout code
2. Setup Node.js 22
3. Install dependencies
4. Setup PostgreSQL test database
5. Run Prisma migrations
6. Execute `npm test`
7. Upload coverage to Codecov

**What this means:** Every push is tested. If tests fail, merge is blocked.

### 2. Deploy Workflow (deploy.yml)
Triggered on: Merge to main branch only

Steps:
1. Checkout code
2. Deploy to Vercel using API token
3. Vercel auto-detects Next.js and runs `npm run build`
4. Production deployment lives

**What this means:** Merging to main = instant live production deployment.

## Local Development vs Production

| Command | Local | Production |
|---------|-------|------------|
| `npm run dev` | Hot reload, local database | N/A |
| `npm run build` | Build Next.js | Run by Vercel |
| `npm test` | Local tests | Run by GitHub Actions |
| `git push` | Nothing | Triggers test workflow |
| Merge to main | Nothing | Triggers deploy workflow |

## Troubleshooting

### Test workflow fails
- Check error message in GitHub Actions tab
- Common causes:
  - Database migration failed
  - Missing dependencies
  - Test syntax errors
- Fix locally: `npm test` and iterate

### Deploy workflow fails
- Usually means tests passed but Vercel build failed
- Check Vercel dashboard for build logs
- Common causes:
  - Environment variable missing
  - Build output incorrect
  - Prisma schema mismatch

### Manual Deployment
Emergency only — not recommended:
```bash
# Prerequisite: vercel CLI installed
npm install -g vercel

# Deploy
vercel --prod

# Vercel will:
# 1. Download environment variables
# 2. Run `npm run build`
# 3. Deploy to production URL
```

---

For more details, see:
- `.github/workflows/test.yml` — Test pipeline
- `.github/workflows/deploy.yml` — Deploy pipeline
```

This documents the pipeline for future team members or troubleshooting.
