Create Prisma schema file with the following models:

1. **datasource** block:
   - provider: "postgresql"
   - url: process.env.DATABASE_URL

2. **generator** block:
   - provider: "prisma-client-js"

3. **model Product**:
   - id: String @id @default(cuid())
   - name: String @unique
   - sku: String @unique
   - description: String? (nullable)
   - unit_price: Float
   - quantity_on_hand: Int @default(0)
   - reserved: Int @default(0) (for pending orders)
   - available: Int @default(0) (computed: on_hand - reserved)
   - created_at: DateTime @default(now())
   - updated_at: DateTime @updatedAt
   - inventory_history: InventoryHistory[]

4. **model InventoryHistory**:
   - id: String @id @default(cuid())
   - product_id: String
   - product: Product @relation(fields: [product_id], references: [id], onDelete: Cascade)
   - previous_quantity: Int
   - new_quantity: Int
   - change_reason: String ("MANUAL_UPDATE" | "ORDER_PLACED" | "ORDER_CANCELLED")
   - changed_by: String (admin user identifier)
   - created_at: DateTime @default(now())

Per D-01: Use PostgreSQL ACID transactions. Per D-03: Single product-level inventory (no locations). Per D-08: No soft deletes.

Indexes: Add @index on product_id in InventoryHistory for fast lookups.

```typescript
import { prisma } from '@/lib/db';

async function main() {
  // Sample products from the business
  const products = [
    { name: '2mm P', sku: '2MM_P', description: 'Plastic roll 2mm', unit_price: 5000, quantity_on_hand: 150 },
    { name: '3mm P', sku: '3MM_P', description: 'Plastic roll 3mm', unit_price: 6000, quantity_on_hand: 450 },
    { name: '4mm P', sku: '4MM_P', description: 'Plastic roll 4mm', unit_price: 7000, quantity_on_hand: 100 },
    { name: '3m Mineral', sku: '3M_MINERAL', description: '3m mineral', unit_price: 15000, quantity_on_hand: 50 },
    { name: '2mm F', sku: '2MM_F', description: 'Film roll 2mm', unit_price: 4500, quantity_on_hand: 100 },
    { name: 'Primer', sku: 'PRIMER', description: 'Primer coat', unit_price: 3000, quantity_on_hand: 50 },
  ];

  for (const product of products) {
    await prisma.product.upsert({
      where: { sku: product.sku },
      update: { quantity_on_hand: product.quantity_on_hand },
      create: product,
    });
  }

  console.log('Seeded sample products');
}

main()
  .then(async () => {
    await prisma.$disconnect();
  })
  .catch(async (e) => {
    console.error(e);
    await prisma.$disconnect();
    process.exit(1);
  });
```

Add to package.json scripts:
"prisma:seed": "ts-node src/db/seed.ts"

Per D-03: Store at product level only (no multi-location complexity).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| App → Database | Untrusted inventory updates from orders |
| Admin → App | Admin manual updates could be malicious |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | Inventory Update API | mitigate | Validate quantity changes in database trigger + application-level range checks (qty >= 0) |
| T-01-02 | Repudiation | Inventory Changes | mitigate | All changes logged to InventoryHistory with timestamp, changed_by, and reason (audit trail) |
| T-01-03 | Information Disclosure | Inventory Levels | mitigate | API endpoints require authentication (defer to Phase 2) |
| T-01-04 | Denial of Service | Database | accept | Low-volume inventory system; DOS not a primary concern for small business |
| T-01-05 | Elevation of Privilege | Admin Inventory Update | mitigate | Check admin role before allowing manual updates (defer admin auth to Phase 2) |
| T-01-06 | Elevation of Privilege | Database | accept | PostgreSQL user has minimal permissions; application user cannot drop tables |

</threat_model>
